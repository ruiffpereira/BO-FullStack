# Reconhecimento profundo — Ginásio (UX · interação · lógica)

**Data:** 2026-07-21 · **Método:** 4 agentes de auditoria em paralelo (BO Ginásio · BO Mensalidades · API gym · PWA gymnoprado), leitura integral do código, achados com `ficheiro:linha`.
**Total: ~179 achados — 20 alta · 66 média · ~93 baixa.**

## SÍNTESE CRUZADA (ler primeiro)

### Temas estruturais (aparecem em ≥2 frentes)

1. **Perda de dados do utilizador** — o tema mais grave, presente nas 4 frentes:
   - BO: "Mudar plano" cria programa inativo invisível (e não há UI para ativar); save do `ClientePlanoEditor` apaga treinos legados (2/dia, multi-dia); editar exercício apaga a media (`media: []` hardcoded).
   - PWA: registo do treino sem `clientUuid` (retry = duplicado), sem fila offline (sem rede = sessão perdida), dots/terminar durante descanso descartam a série feita; qualquer erro no restauro do perfil apaga o token.
   - API: hard-delete de pagamentos/clientes reescreve receita histórica (contra a regra "nunca apagar histórico").

2. **Derivações financeiras divergentes** — "em atraso" tem 3 definições (API getFinance vs cockpit vs ficha), "MRR" tem 3 (finance vs analytics vs dashboard), "por cobrar" tem 2, "recebido" ignora parciais até virarem `paid`. Fix único: empurrar TODAS as derivações para a API (fonte única, padrão `apptStatusView`) e o cliente só formata.

3. **Contrato Kubb/swagger em drift** — 73 `as any` em Ginasio.tsx, tipos locais reescritos à mão em GymMensalidade.tsx, `mode`/`setRows` fora do spec da PWA, swagger sem `vatRate`/`payOnly`/`membershipStatus`/`createdAt`. Fix upstream: completar swagger → `kubb:refresh` nos 2 frontends → limpar casts.

4. **Zod ≈ 2% no gym** — 1 endpoint em ~45 (`inviteMember`, que é o padrão-ouro a replicar). Writes de dinheiro (`upsertPayment`, subscrições) e `createLog` da PWA aceitam NaN/negativos/datas inválidas → 500s.

5. **Relógio fragmentado** — UTC (`toISOString`) vs hora local em ≥5 sítios (controllers, cron reminders, calendar, BO `currentPeriod`, PWA dashboard). Na fronteira do dia/mês em horário de verão PT, mês corrente/overdue/streak/dots divergem. Fix: helper único Europe/Lisbon.

6. **Disciplina de plataforma no BO** — write-guard ausente (Ginasio.tsx: zero; GymMensalidade: metade), `Select` nativo vs regra Combobox, confirmações hand-rolled vs `ConfirmDialog`, e `Modal` partilhado (`ui.jsx`) sem focus-trap + **Esc fecha a pilha inteira de modais** (fix no ui.jsx beneficia a app toda).

7. **Código morto com funcionalidades órfãs** — ~530 linhas (~17%) em Ginasio.tsx; o `ProgramasTab` morto levou consigo a ÚNICA UI de ativar/eliminar/listar programas — raiz direta do bug nº1.

8. **i18n/CMS incompleto no client-facing** — PDF do plano 100% PT hardcoded (material para o cliente final); PWA com saudações/dias da semana/install prompt fora do CMS; `dayLabel` "Dia N" e métodos de pagamento PT persistidos no modelo (contra "tokens EN no modelo").

### Ordem de ataque proposta

- **Fase 1 — Estancar perda de dados (alta, ~semana):** assign ativa/pergunta + UI mínima de programas no ClienteProgresso · save do ClientePlanoEditor seguro (detetar/suportar >1 treino/dia e multi-dia) · fix media wipe · PWA: clientUuid + finishRest antes de navegar/gravar + fila offline persistida + logout só em 401/403 + confirmações destrutivas.
- **Fase 2 — Números que batem certo:** derivação financeira única na API (overdue do mês automático, órfãos de subscrição, parciais no recebido, MRR único) + relógio Europe/Lisbon + fix seleção bulk + transações (programa ativo, upsertPayment).
- **Fase 3 — Fundações de contrato:** swagger completo → kubb:refresh (BO + PWA) → matar os `as any`/tipos manuais → apagar as ~530 linhas mortas → zod nos writes (padrão inviteMember) → perf do /stats (carregar logs 1×).
- **Fase 4 — Interação:** modal stack + focus-trap no ui.jsx · write-guard completo · ConfirmDialog em todo o lado · dirty-check nos editores full-screen · PWA: alvos de toque ≥44px, vibração/som no fim do descanso, steppers com input direto.
- **Fase 5 — i18n/CMS:** PDF multilíngua (via localizeProgramDTOs + chaves CMS) · strings PWA para o CMS · tokens EN no modelo (dayLabel, métodos de pagamento — com migração).
